# 代码修改规范

本文件用于约束 AI 或开发者修改本项目代码时的统一风格。修改代码前必须先阅读项目现有实现，优先沿用当前目录结构、命名方式、依赖注入方式和前后端交互方式。

## 一、项目技术栈与目录边界

本项目由三部分组成：

- `backend/`：FastAPI + SQLAlchemy + Pydantic 后端服务。
- `vue/`：Vue 3 + TypeScript + Vite + Pinia + Ant Design Vue 前端。
- `database/`：MySQL 建表、初始化和更新 SQL 脚本。

修改时必须遵守以下边界：

- 后端接口只写在 `backend/app/api/routes/`。
- 后端请求与响应模型只写在 `backend/app/schemas/`。
- 后端业务逻辑只写在 `backend/app/services/`。
- 后端数据库 ORM 模型只写在 `backend/app/models/`。
- 后端外部模型/API 调用只写在 `backend/app/providers/`。
- 前端接口封装只写在 `vue/src/api/`。
- 前端通用请求逻辑只写在 `vue/src/utils/request.ts`。
- 前端页面只写在 `vue/src/views/`。
- 数据库结构变化必须同步更新 `database/` 下的 SQL 或 Alembic 迁移。

不要把数据库查询、积分结算、外部 API 调用等业务逻辑直接写进路由或 Vue 页面。

## 二、统一接口返回结构

后续新增或重构接口时，后端统一返回以下结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "success": true
}
```

字段含义：

- `code`：业务状态码。`0` 表示成功，非 `0` 表示业务失败。
- `message`：提示信息。成功默认使用 `"success"`，失败返回明确中文错误原因。
- `data`：真实业务数据。没有数据时返回 `null`，列表接口返回分页对象。
- `success`：是否成功。成功为 `true`，失败为 `false`。

推荐后端公共响应模型：

```python
from typing import Generic, Optional, TypeVar
from pydantic import BaseModel
from pydantic.generics import GenericModel

T = TypeVar("T")


class ApiResponse(GenericModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: Optional[T] = None
    success: bool = True
```

推荐封装方法：

```python
def success_response(data=None, message: str = "success"):
    return {
        "code": 0,
        "message": message,
        "data": data,
        "success": True
    }
```

失败返回建议继续使用 `HTTPException` 表达 HTTP 状态，但响应体也应尽量统一成：

```json
{
  "code": 40001,
  "message": "错误原因",
  "data": null,
  "success": false
}
```

新增全局异常处理时，应统一处理：

- `HTTPException`
- Pydantic 参数校验错误
- 未捕获异常
- 业务异常

不要在不同接口中混用 `{"message": "..."}`、直接对象、直接数组、`{"detail": "..."}` 等多种成功返回格式。历史接口可以逐步迁移，但新增接口必须使用统一结构。

## 三、分页返回结构

列表接口的 `data` 统一使用以下格式：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

约束：

- 页码参数统一命名为 `page`。
- 每页数量统一命名为 `page_size`。
- `page` 默认 `1`，最小 `1`。
- `page_size` 默认 `20`，最大 `100`。
- 后端使用 `Query` 设置参数边界。

## 四、后端代码风格

### 1. 路由层

路由层只负责：

- 接收请求参数。
- 注入 `db`、当前用户等依赖。
- 调用 service。
- 组装统一响应。
- 抛出明确的 HTTP 或业务异常。

路由层不要直接写复杂 SQL、积分结算、外部 API 调用或多步骤业务流程。已有少量历史代码可以逐步迁移，新代码必须优先放到 service/provider。

示例：

```python
@router.get("/{task_id}", response_model=ApiResponse[TaskDetailResponse])
def get_task_detail(
    task_id: str,
    current_user: User = Depends(get_current_user),
    db: Session = Depends(get_db)
):
    task = generation_service.get_user_task_detail(
        db=db,
        user_id=current_user.id,
        task_id=task_id
    )
    return success_response(data=task)
```

### 2. Service 层

Service 层负责业务规则：

- 参数业务校验。
- 数据库查询与写入。
- 事务提交与回滚。
- 积分冻结、扣除、退回。
- 任务状态流转。

要求：

- 写入数据库时必须明确事务边界。
- 出现异常时必须 `db.rollback()`。
- 重要业务节点必须写日志。
- 不要吞掉异常后返回假成功。
- 不要把当前登录用户 ID 从前端传入，必须从 token 解析。

### 3. Schema 层

Pydantic 模型命名规则：

- 请求模型：`XxxRequest`
- 响应数据模型：`XxxResponse`
- 列表响应数据：`XxxListResponse`
- 通用响应包裹：`ApiResponse[T]`

字段命名优先沿用后端现状，接口 JSON 使用小写下划线，例如：

- `task_id`
- `price_config_id`
- `page_size`
- `created_at`

不要在同一接口中混用 `camelCase` 和 `snake_case`。

### 4. ORM 模型

ORM 模型只描述数据库结构，不写业务逻辑。

新增字段时必须同步考虑：

- 数据库 SQL 或 Alembic 迁移。
- Pydantic schema。
- service 查询和写入逻辑。
- 前端类型和展示。

### 5. 日志

后端统一使用 `app.utils.logger.app_logger`。

日志要求：

- 成功的关键业务节点使用 `info`。
- 可恢复或重复操作使用 `warning`。
- 异常失败使用 `error`。
- 日志中可以记录 `task_id`、`user_id`、模型名、积分数量等排查信息。
- 不要记录密码、token、API key、完整密钥、隐私图片内容。

## 五、前端代码风格

### 1. API 封装

所有 HTTP 请求必须通过 `vue/src/utils/request.ts` 中的 axios 实例发起。

业务 API 必须写在 `vue/src/api/`，不要在页面组件里直接写：

```ts
axios.get(...)
fetch(...)
```

API 函数命名使用动词开头：

- `login`
- `register`
- `getUserInfo`
- `createImageTask`
- `getTaskDetail`
- `getMyTasks`

### 2. 统一响应处理

当前 `request.ts` 直接返回 `response.data`。迁移到统一返回结构后，建议拦截器统一处理：

```ts
request.interceptors.response.use(
  (response) => {
    const res = response.data

    if (res?.success === false || res?.code !== 0) {
      message.error(res?.message || '请求失败')
      return Promise.reject(res)
    }

    return res?.data
  },
  (error) => {
    return Promise.reject(error)
  }
)
```

页面组件只关心业务数据，不直接解析 `code`、`success`、`message`。错误提示优先由请求拦截器统一处理。

### 3. TypeScript 类型

新增 API 时必须补充请求参数和响应数据类型。

推荐写法：

```ts
export interface CreateImageTaskParams {
  price_config_id: number
  prompt: string
}

export interface TaskCreateResult {
  task_id: string
  status: string
  frozen_points: number
  error_message?: string | null
}

export function createImageTask(data: CreateImageTaskParams) {
  return request.post<TaskCreateResult>('/image/generate', data)
}
```

不要在多个页面重复定义同一份接口类型。

### 4. Vue 组件

Vue 页面只负责：

- 用户交互。
- 调用 API。
- 展示状态。
- 表单校验。
- 路由跳转。

不要在页面中写后端业务规则，例如积分结算规则、任务状态强制修改、价格计算规则等。

状态管理优先放在 Pinia store 中，例如用户 token、用户信息、跨页面共享状态。

## 六、错误处理规范

后端错误信息必须明确，避免只返回 `"error"` 或 `"failed"`。

推荐业务错误码分段：

- `0`：成功
- `400xx`：请求参数或业务校验错误
- `401xx`：登录、token、认证错误
- `403xx`：权限错误
- `404xx`：资源不存在
- `409xx`：状态冲突、重复操作
- `500xx`：服务端或第三方服务错误

示例：

```json
{
  "code": 40401,
  "message": "任务不存在",
  "data": null,
  "success": false
}
```

第三方 API 错误必须转换成项目内部错误，不要把第三方原始错误结构直接透传给前端。

## 七、安全规范

必须遵守：

- API 中转站密钥只能放后端环境变量，不能写进前端。
- JWT 只能通过 `Authorization: Bearer <token>` 传递。
- 前端不能传 `user_id`，后端必须从 token 中解析当前用户。
- 密码只允许存储哈希值，不能明文存储或打印日志。
- 不要把 `.env`、密钥、token、私有 URL 提交到仓库。
- 管理员接口必须单独校验管理员身份，不能只依赖普通登录态。

## 八、数据库修改规范

涉及数据库结构变化时，必须同步处理：

1. 修改 ORM 模型。
2. 增加 SQL 更新脚本或 Alembic 迁移。
3. 更新初始化 SQL。
4. 更新相关 schema。
5. 更新 service 读写逻辑。
6. 如影响前端展示，同步更新前端类型和页面。

积分、订单、任务状态相关修改必须考虑并发安全：

- 使用事务。
- 必要时使用行锁。
- 防止重复结算。
- 防止重复扣减积分。

## 九、代码修改流程

AI 修改代码时必须按以下流程执行：

1. 先阅读相关文件，不凭空改代码。
2. 判断修改影响范围，尽量做小范围变更。
3. 优先复用项目已有工具函数、service、schema 和请求封装。
4. 修改后检查前后端字段名是否一致。
5. 涉及接口返回时检查是否符合统一响应结构。
6. 涉及数据库时检查 SQL、ORM、schema、service 是否同步。
7. 能运行测试或构建时，应运行对应命令验证。
8. 最终说明改了什么、验证了什么、还有什么风险。

不要做以下事情：

- 不要随意重构无关文件。
- 不要删除用户已有代码。
- 不要引入项目未使用的大型依赖。
- 不要改变公开接口字段名，除非同步修改所有调用方。
- 不要把临时调试代码、测试 token、真实密钥留在仓库。
- 不要绕过 `request.ts`、service 层或依赖注入体系。

## 十、提交前检查清单

修改完成后至少检查：

- 后端新增接口是否有 schema。
- 后端响应是否统一为 `ApiResponse` 结构。
- 错误信息是否明确。
- 前端 API 是否统一放在 `src/api/`。
- 前端是否通过 `request.ts` 请求。
- 前后端字段名是否一致。
- 数据库变更是否有 SQL 或迁移。
- 是否没有泄露 token、密码、API key。

本规范优先级高于临时生成代码的个人风格。若现有代码与本规范不一致，新增代码按本规范执行，历史代码在相关功能改造时逐步迁移。
