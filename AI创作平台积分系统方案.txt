AI创作平台积分系统方案
========================

一、项目目标
------------

当前系统需要增加以下能力：

1. 用户登录和注册
2. 用户充值兑换积分
3. 用户使用积分调用生图模型
4. 后端统一使用平台自己的 API 中转站密钥调用模型
5. API 中转站余额由平台自己维护，不由项目自动充值
6. 不同模型、图片尺寸、生成数量对应固定积分
7. 用户生成任务成功扣积分，失败退积分

核心逻辑：

用户注册/登录
    ↓
用户充值
    ↓
充值成功后获得积分
    ↓
用户选择固定模型规格生成图片
    ↓
后端根据模型配置计算固定积分
    ↓
检查用户积分是否足够
    ↓
冻结积分
    ↓
调用 API 中转站
    ↓
成功扣积分
    ↓
失败退积分


二、整体系统架构
----------------

前端：Vue3 + Ant Design Vue
后端：FastAPI
数据库：MySQL
模型调用：统一走 API 中转站
支付：支付宝 / 微信 / Stripe 等，可按实际情况选择

整体模块：

1. 用户认证模块
   - 注册
   - 登录
   - 获取当前用户信息
   - JWT 鉴权

2. 积分账户模块
   - 查询积分余额
   - 查询积分流水
   - 充值增加积分
   - 生图冻结积分
   - 成功扣除积分
   - 失败退回积分

3. 充值模块
   - 充值套餐
   - 创建充值订单
   - 支付回调
   - 支付成功后增加积分

4. 模型价格模块
   - 配置模型
   - 配置尺寸
   - 配置生成数量
   - 配置固定消耗积分

5. 生图任务模块
   - 创建生图任务
   - 调用 API 中转站
   - 记录任务状态
   - 保存返回结果
   - 成功扣积分
   - 失败退积分

6. 后台管理模块
   - 用户管理
   - 充值套餐管理
   - 模型价格管理
   - 充值订单管理
   - 积分流水查询
   - 生图任务查询


三、核心业务规则
----------------

1. 用户必须登录后才能充值、查询积分、生成图片。
2. 注册用户时，系统自动创建积分账户。
3. 前端不能传 user_id，后端必须从 JWT token 中解析当前用户。
4. 前端不能传积分价格，后端必须根据 price_config_id 查询数据库中的价格配置。
5. 生图任务创建前必须先检查积分是否足够。
6. 生图任务开始前先冻结积分。
7. 生图任务成功后，从冻结积分中正式扣除。
8. 生图任务失败后，把冻结积分退回可用积分。
9. 所有积分变动都必须写入积分流水表。
10. 支付回调必须做幂等处理，防止重复加积分。
11. API 中转站密钥只能放在后端，不能暴露给前端。
12. API 中转站余额由平台自己维护，项目不负责自动给中转站充值。


四、用户体系设计
----------------

1. 用户注册

接口：

POST /api/auth/register

请求示例：

{
  "username": "hgg",
  "password": "123456",
  "confirm_password": "123456"
}

处理流程：

1. 校验 username 是否为空
2. 校验 password 是否符合规则
3. 校验 confirm_password 是否一致
4. 判断 username 是否已存在
5. 对密码进行 hash 加密
6. 创建 users 记录
7. 创建 user_point_accounts 记录
8. 返回注册成功

注册时必须使用数据库事务：

开启事务
    ↓
创建用户
    ↓
创建积分账户
    ↓
提交事务

这样可以避免用户创建成功但积分账户创建失败。


2. 用户登录

接口：

POST /api/auth/login

请求示例：

{
  "username": "hgg",
  "password": "123456"
}

处理流程：

1. 根据 username 查询用户
2. 判断用户是否存在
3. 判断用户状态是否 normal
4. 校验密码 hash
5. 登录成功后生成 JWT access_token
6. 返回 token 和用户信息

返回示例：

{
  "access_token": "xxxxx",
  "token_type": "Bearer",
  "user": {
    "id": 1,
    "username": "hgg",
    "nickname": "hgg"
  }
}

前端后续请求都需要携带：

Authorization: Bearer xxxxx


五、JWT 鉴权设计
----------------

后端 .env 配置：

JWT_SECRET_KEY=change-this-secret-key
JWT_ALGORITHM=HS256
JWT_EXPIRE_MINUTES=10080

说明：

10080 分钟等于 7 天。

JWT 内容示例：

{
  "sub": "1",
  "username": "hgg",
  "exp": 过期时间
}

所有需要登录的接口，都需要通过 token 获取当前 user_id。

不要让前端传 user_id。

错误请求示例：

{
  "user_id": 1,
  "price_config_id": 1,
  "prompt": "xxx"
}

正确请求示例：

{
  "price_config_id": 1,
  "prompt": "xxx"
}


六、数据库表设计
----------------

1. 用户表 users

CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    username VARCHAR(64) DEFAULT NULL COMMENT '用户名',
    email VARCHAR(128) DEFAULT NULL COMMENT '邮箱',
    phone VARCHAR(32) DEFAULT NULL COMMENT '手机号',

    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',

    nickname VARCHAR(64) DEFAULT NULL COMMENT '昵称',
    avatar_url VARCHAR(255) DEFAULT NULL COMMENT '头像',

    status VARCHAR(32) NOT NULL DEFAULT 'normal' COMMENT 'normal, disabled',

    last_login_at DATETIME DEFAULT NULL,
    last_login_ip VARCHAR(64) DEFAULT NULL,

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_username (username),
    UNIQUE KEY uk_email (email),
    UNIQUE KEY uk_phone (phone),
    INDEX idx_status (status)
) COMMENT='用户表';


2. 用户积分账户表 user_point_accounts

CREATE TABLE user_point_accounts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL UNIQUE COMMENT '用户ID',

    balance_points BIGINT NOT NULL DEFAULT 0 COMMENT '可用积分',
    frozen_points BIGINT NOT NULL DEFAULT 0 COMMENT '冻结积分',

    total_recharged_points BIGINT NOT NULL DEFAULT 0 COMMENT '累计充值积分',
    total_consumed_points BIGINT NOT NULL DEFAULT 0 COMMENT '累计消费积分',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_user_id (user_id)
) COMMENT='用户积分账户表';


字段说明：

balance_points：用户当前可用积分
frozen_points：用户任务进行中被冻结的积分
total_recharged_points：累计充值获得积分
total_consumed_points：累计实际消耗积分


3. 积分流水表 point_transactions

CREATE TABLE point_transactions (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    transaction_no VARCHAR(64) NOT NULL UNIQUE COMMENT '积分流水号',
    user_id BIGINT NOT NULL COMMENT '用户ID',

    type VARCHAR(32) NOT NULL COMMENT '类型: recharge, freeze, consume, refund, unfreeze, admin_adjust',
    direction VARCHAR(16) NOT NULL COMMENT '方向: income, expense, freeze, unfreeze',

    points BIGINT NOT NULL COMMENT '变动积分',

    balance_before BIGINT NOT NULL COMMENT '变动前可用积分',
    balance_after BIGINT NOT NULL COMMENT '变动后可用积分',

    frozen_before BIGINT NOT NULL DEFAULT 0 COMMENT '变动前冻结积分',
    frozen_after BIGINT NOT NULL DEFAULT 0 COMMENT '变动后冻结积分',

    related_order_no VARCHAR(64) DEFAULT NULL COMMENT '关联充值订单号',
    related_task_id VARCHAR(64) DEFAULT NULL COMMENT '关联生成任务ID',

    remark VARCHAR(255) DEFAULT NULL,

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_user_id_created_at (user_id, created_at),
    INDEX idx_related_order_no (related_order_no),
    INDEX idx_related_task_id (related_task_id)
) COMMENT='积分流水表';


流水类型说明：

recharge：充值到账
freeze：创建任务时冻结积分
consume：任务成功后正式扣除积分
unfreeze：任务失败后解冻积分
refund：退款或补偿
admin_adjust：后台人工调整积分

八、积分扣费流程
----------------

1. 创建任务前

用户可用积分：1000
用户冻结积分：0
本次生图需要：80 积分


2. 创建任务时

可用积分减少 80
冻结积分增加 80

结果：

可用积分：920
冻结积分：80


3. 任务成功时

冻结积分减少 80
累计消费增加 80

结果：

可用积分：920
冻结积分：0
累计消费：+80


4. 任务失败时

冻结积分减少 80
可用积分增加 80

结果：

可用积分：1000
冻结积分：0


九、创建生图任务核心流程
------------------------

后端处理流程：

1. 用户携带 JWT token 请求 /api/image/generate
2. 后端从 token 中解析 user_id
3. 根据 price_config_id 查询 model_price_configs
4. 判断模型配置是否存在、是否启用、是否为 image 类型
5. 获取本次需要消耗的 points
6. 开启数据库事务
7. 使用 SELECT ... FOR UPDATE 锁定 user_point_accounts
8. 判断 balance_points 是否足够
9. 如果不足，返回“积分不足”
10. 如果足够，扣减 balance_points，增加 frozen_points
11. 写入 point_transactions 冻结流水
12. 创建 generation_tasks 任务记录
13. 提交事务
14. 后端使用 API 中转站 Key 调用模型
15. 如果模型调用成功：
    - 更新任务状态为 success
    - 减少 frozen_points
    - 增加 total_consumed_points
    - 写 consume 流水
    - 保存 result_json
16. 如果模型调用失败：
    - 更新任务状态为 failed
    - 减少 frozen_points
    - 增加 balance_points
    - 写 unfreeze 流水
    - 保存 error_message


十、API 中转站调用方案
---------------------

后端 .env 配置：

API_GATEWAY_BASE_URL=https://xxx.com/v1
API_GATEWAY_API_KEY=sk-xxxx

调用规则：

1. 所有模型请求统一由后端调用 API 中转站
2. 前端不能直接请求 API 中转站
3. 前端不能看到 API Key
4. API Key 只放在服务器环境变量里
5. API 中转站余额由平台自己维护
6. 项目不需要做 API 中转站自动充值功能

十二、前端登录状态处理
----------------------

登录成功后，前端保存 access_token。

推荐保存到 localStorage。

Axios 请求拦截器示例：

axios.interceptors.request.use((config) => {
  const token = localStorage.getItem('access_token')

  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }

  return config
})

如果接口返回 401：

1. 清除本地 token
2. 跳转登录页
3. 提示用户重新登录


十三、安全要求
--------------

1. 密码不能明文保存，必须保存 password_hash。
2. 推荐使用 bcrypt 加密密码。
3. JWT_SECRET_KEY 不能写死在代码里，必须放到 .env。
4. API 中转站 Key 不能返回给前端。
5. 前端不能传 user_id。
6. 前端不能传积分价格。
7. 后端必须校验 price_config_id 是否存在并启用。
8. 支付回调必须校验支付平台签名。
9. 支付回调必须幂等，避免重复加积分。
10. 积分扣减必须使用事务和行锁。
11. 生成任务只能查询当前用户自己的任务。
12. 管理员接口和普通用户接口需要分开鉴权。


十四、支付回调幂等处理
----------------------

支付平台可能会重复通知，所以必须做幂等。

处理流程：

1. 根据 order_no 查询充值订单
2. 如果订单不存在，返回错误
3. 如果订单已经是 paid，直接返回成功，不重复加积分
4. 如果订单不是 paid：
   - 校验支付状态
   - 更新订单为 paid
   - 设置 paid_at
   - 给用户增加积分
   - 写 recharge 积分流水
5. 提交事务


十五、并发控制
--------------

为什么需要冻结积分？

假设用户有 100 积分，单次生图需要 80 积分。

用户快速点击两次：

请求 A 需要 80 积分
请求 B 也需要 80 积分

如果不使用事务和锁，两个请求可能都判断余额足够，导致超扣。

正确方式：

请求 A：
    锁定账户
    可用积分 100 → 20
    冻结积分 0 → 80

请求 B：
    锁定账户
    发现可用积分只有 20
    返回积分不足

数据库中使用：

SELECT * FROM user_point_accounts
WHERE user_id = ?
FOR UPDATE;

十七、最终结论
--------------

当前项目推荐采用以下方案：

1. 用户先注册登录。
2. 注册时自动创建积分账户。
3. 用户充值只兑换平台内部积分。
4. 项目不负责给 API 中转站自动充值。
5. API 中转站余额由平台自己长期维护充足。
6. 所有模型调用都统一走平台自己的 API 中转站 Key。
7. 生图模型、图片尺寸、图片数量都通过后台固定配置。
8. 每个模型规格对应固定积分。
9. 前端只展示后台允许的模型规格。
10. 前端创建生图任务时只传 price_config_id 和 prompt。
11. 后端根据 price_config_id 查询真实模型、尺寸、数量和积分。
12. 创建任务前冻结积分。
13. 任务成功后正式扣除积分。
14. 任务失败后退回冻结积分。
15. 所有积分变化必须写入流水。
16. 所有充值、积分、任务都必须绑定 users.id。
17. 前端不能传 user_id，后端必须从 token 中获取当前用户。
